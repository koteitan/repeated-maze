# hs2maze — ステートマシンから迷路への変換器

Haskell 風のステートマシン定義を繰り返し迷路のポート文字列に変換する。
任意の 2レジスタ ミンスキーマシン（または類似のステートマシン）を
繰り返し迷路として符号化する体系的な手段を提供する。

## 使い方

```bash
python3 hs2maze.py [input.hs]          # 迷路文字列を stdout に出力
python3 hs2maze.py input.hs 2>/dev/null | xargs -I{} \
  ../../repeated-maze solve '{}' --bfs -v  # 解いて検証
```

ファイルを指定しない場合は stdin から読む。診断情報は stderr に出力。

## 入力形式

```haskell
myfunc :: (Int, Int, Int) -> (Int, Int, Int)
myfunc (x, y, 0) = myfunc (x+1, y,   2)  -- x+1
myfunc (x, y, 2) = myfunc (x,   y+3, 3)  -- y+3
myfunc (x, y, 3) = myfunc (x-1, y,   1)  -- x-1, ゴール到達
```

各行は遷移 `(x, y, pc_src) → (x', y', pc_dst)` を定義する：

- `x`, `y`: 座標変数。`x+k`, `x-k`, `y+k`, `y-k`, `_`（変化なし）が使える。
- `pc`: プログラムカウンタ（整数リテラル）。
- 各行で dx, dy のうち**片方のみ**非ゼロ。
- `--` コメントと `::` 型シグネチャは無視される。

### 予約済み PC 値

| PC | 役割 |
|---|---|
| 0 | 開始状態。W0@(1,1) = E0@(0,1) に対応。 |
| 1 | ゴール状態。W1@(1,1) = E1@(0,1) に対応。 |
| 2+ | ユーザー定義状態。 |

迷路に解が存在するためには、ステートマシンが pc=1 に到達する時点で座標が (0, 1) に戻っている必要がある。

## ミンスキーレジスタマシン

**ミンスキーレジスタマシン**（カウンタマシン）は以下で構成される：

- 有限個の**状態**（プログラムカウンタ値） pc ∈ {0, 1, ..., n}
- 2つの非負整数**レジスタ** x, y
- 各状態は以下のいずれかの命令を持つ：

| 命令 | 効果 |
|---|---|
| INC(r, pc') | レジスタ r を 1 増加、状態 pc' へ遷移 |
| DEC(r, pc', pc'') | r > 0 なら r を減少し pc' へ。r = 0 なら pc'' へ |
| HALT | 実行停止 |

ミンスキーは 2レジスタマシンが**チューリング完全**であることを証明した。
つまり任意の計算可能関数を計算できる。
これは繰り返し迷路の最短経路問題が一般に決定不能であることを意味する。

### hs2maze 形式での符号化

INC と無条件ジャンプは遷移に直接対応する：

```haskell
-- INC x: x += 1
machine (x, y, 3) = machine (x+1, y, 4)

-- INC y: y += 1
machine (x, y, 5) = machine (x, y+1, 6)

-- DEC x（非ゼロ分岐）: x -= 1
machine (x, y, 7) = machine (x-1, y, 8)
```

**ゼロテスト**（DEC のゼロ分岐）には nx/ny 境界ブロックが必要だが、
現在の hs2maze は nx/ny ポートを生成しない。ゼロテストが必要な場合は
手動で nx/ny ポートを追加するか、デクリメント時にレジスタがゼロにならない
設計にする（[カウンターポンプ](../counter-pump/README-ja.md)のように）。

## 変換の原理

### 正準状態の対応

ユーザー状態 `(x, y, pc)` は迷路の正準状態 `(x, y, E, pc)` に対応する：

```
ユーザー状態:  (x, y, pc)
迷路状態:      (x, y, E, pc)   — E型正準状態
物理位置:      E[pc] @ ブロック (x, y)  =  W[pc] @ ブロック (x+1, y)
```

全てのユーザー pc 値は **E/W ターミナルインデックス**の名前空間を使う。
**N/S ターミナルインデックス**の名前空間は y 移動の中間状態に使われる。

### 移動の変換

各ステートマシン遷移はポートチェインに変換される：

#### x+k（k 迷路ステップ）

W→E ホップのチェイン。1ホップごとにブロック境界を1つ東に越える。

```
W[pc₀] → E[t₁],  W[t₁] → E[t₂],  ...,  W[tₖ₋₁] → E[pc_dst]
```

例: pc=0 から pc=2 へ x+1

```
ポート: W0 → E2
パス: (0,1,E0) --[W0→E2]--> (1,1,E2)
```

#### x-k（k 迷路ステップ）

E→W ホップのチェイン。1ホップごとにブロック境界を1つ西に越える。

```
E[pc₀] → W[t₁],  E[t₁] → W[t₂],  ...,  E[tₖ₋₁] → W[pc_dst]
```

#### y+k（k+1 迷路ステップ）

E/W 境界から N/S 境界へ切り替え、戻す必要がある。
中間状態は N/S ターミナルインデックス（E/W とは別の名前空間）を使う。

```
W[pc₀] → N[n₁],   S[n₁] → N[n₂],  ...,  S[nₖ] → W[pc_dst]
 (E→N切替)            (y+1 ホップ)          (N→E切替)
```

パスの経路: 東にブロックに入り、北に k ブロック通過し、西に戻る。
正味の変位: (0, +k)。東向きと西向きの成分が打ち消し合う。

```
  ブロック(x+1, y+k)
      ↑  ← S[nₖ]→W[dst] (西に出る)
      |
  ブロック(x+1, y+1)
      ↑  ← S[n₁]→N[n₂] (北に継続)
      |
  ブロック(x+1, y)
      ↑  ← W[src]→N[n₁] (入って北に出る)
      |
  ブロック(x, y)   ← 開始: E[src] はここ
```

#### y-k（k+1 迷路ステップ）

同じ発想で南向き：

```
W[pc₀] → S[n₁],   N[n₁] → S[n₂],  ...,  N[nₖ] → W[pc_dst]
```

#### noop / dx=0, dy=0（2 迷路ステップ）

東に出て西に戻るバウンス：

```
W[pc₀] → E[tmp],   E[tmp] → W[pc_dst]
```

### ステップコストの一覧

| 移動 | 迷路ステップ数 | ポートチェイン |
|---|---|---|
| x+k | k | W→E × k |
| x-k | k | E→W × k |
| y+k | k+1 | W→N, S→N × (k-1), S→W |
| y-k | k+1 | W→S, N→S × (k-1), N→W |
| noop | 2 | W→E, E→W |

### ターミナルインデックスの割り当て

- **E/W インデックス** 0, 1, ..., max_pc: ユーザー定義の pc 値
- **E/W インデックス** max_pc+1, ...: x チェインと noop の中間値
- **N/S インデックス** 0, 1, ...: y チェインの中間値

出力の nterm = max(max_ew_index + 1, max_ns_index + 1)。

### 全てのエントリが W 側を使う理由

全ての遷移の最初のポートは `W[src]→...` エントリ（ブロック (x+1, y) の西側から入る）を使う。
これにより x=0 の nx 境界ブロック（E→E ポートしか存在しない）を回避する。
x+1 ≥ 1 は常に成り立つため、W 側エントリは常に通常ブロック内にある。

x-k の場合は `E[src]→W[...]` ポートをブロック (x, y) の E 側エントリとして使う。
これは x > 0 を要求するが、ステートマシンが負の x 値に遷移しない限り保証される。
