# Epic 4: 隣接ブロック間ターミナル位置合わせ

## 問題

diagonal2 アルゴリズムでは W と E に別々の行、N と S に別々の列を割り当てている。
そのため、隣接ブロック間で同一物理点であるべきターミナルの座標がずれる。

- E[i]@(x,y) = W[i]@(x+1,y) だが、E チャネル行 ≠ W チャネル行 → y がずれる
- N[i]@(x,y) = S[i]@(x,y+1) だが、N チャネル列 ≠ S チャネル列 → x がずれる

## 方針

ルーティングの最後に**位置合わせセグメント**を追加する。

### E ターミナルへの接続 (y 位置合わせ)

E[i] のグリッド行 (E チャネル行) と、対応する W[i] のグリッド行 (W チャネル行) が異なる。
E ターミナルに到達する直前に、W[i] の行まで V セグメントで移動し、そこから E 境界へ H で接続する。

```
ルート本体 ─── (E チャネル行, mc) ─── 位置合わせ V ─── (W チャネル行, E境界)
                                     ↕ E行→W行         → 隣ブロックの W[i] と一致
```

具体的には:
- 現在の dg (E チャネル位置) を「内部終点」とする
- 実際の E 境界位置は W[i] と同じ行にする
- ルート末尾に V セグメント (E チャネル行 → W チャネル行) を追加
- 最後に H セグメントで E 境界まで移動

### S ターミナルへの接続 (x 位置合わせ)

S[i] のグリッド列 (S チャネル列) と、対応する N[i] のグリッド列 (N チャネル列) が異なる。
S ターミナルに到達する直前に、N[i] の列まで H セグメントで移動し、そこから S 境界へ V で接続する。

```
ルート本体 ─── (mr, S チャネル列) ─── 位置合わせ H ─── (S境界行, N チャネル列)
                                     ↔ S列→N列         → 隣ブロックの N[i] と一致
```

### 位置合わせセグメントの衝突

位置合わせ V/H セグメントは E 境界列 / S 境界行の近くに配置される。
他のルートの位置合わせセグメントと重なる可能性がある。

**解決策**: 各 E/S ターミナルの位置合わせ用に専用の列/行を確保する。
- E 位置合わせ: E チャネルごとに専用列 (E 境界の直前)
- S 位置合わせ: S チャネルごとに専用行 (S 境界の直前)

または、位置合わせ V は E 境界列上で行う (E 境界は各ブロック共通なので衝突しない)。
→ E 境界列に複数の V セグメントが乗る → 同方向重なりの可能性あり。

**より安全な方法**: 位置合わせを dg のチャネル列/行で行う:
- E[i] の位置合わせ V: dg の列 (E 境界列 = nC-1) で dg.r → W チャネル行
  → 全 E ターミナルが同じ列 (E 境界) だが、各 V の行範囲は E チャネル行〜W チャネル行。
  E チャネル行は各ポート端で一意なので、V 同士の行範囲が重なる可能性あり。

**最も安全な方法**: 位置合わせ V セグメントを各ルートの専用中間列で行い、最後に W 行で E 境界まで H:
- ルート → (dg_E行, mc) → V(dg_E行→dg_W行, mc) → H(dg_W行, mc→E境界)
  ただし mc は Z-bend の中間列。L-bend では mc がない → L-bend にも中間列が必要になる。

**簡潔な方法**: ルートの dg を E チャネル行ではなく W チャネル行に設定する。
つまり E ポート端の grid position を W チャネル行に揃える:

```
mkChG for E: row = W と同じ行 (ch + 1)、col = nC - 1
```

→ これにより E と W が同じ行を共有する。ただし diagonal2 の「W と E は別行」という
衝突回避の根拠が崩れる。

## ボツ: W/E 行共有案

tester の検証で 3 つの反例が見つかった (epic/4/test/review-report.md)。
W/E 行共有は L-bend の H 重なりを引き起こすため不採用。

## 採用する方法

**ルーティングは diagonal2 のまま (別行/別列)。表示レベルで位置合わせ。**

diagonal2 の衝突回避は「中間セグメントが専用行/列」に依存しており、
sg/dg の行/列が共有されても中間セグメントが一意なら衝突しない。

L-bend の衝突: sg 行と dg 列の交点で折れる。sg 行が共有されると H 重なりの可能性。
→ **L-bend にも専用中間列/行を割り当てて Z-bend 化する。**

全ポートを Z-bend 化すれば、sg/dg 行/列が共有されても安全:
- H1 は sg の行 (共有可能だが、H1 の範囲は sg.c〜mc で、mc がポートごとに異なる)
  → H1 同士は同じ行にあるが、mc が異なるので範囲が異なる
  → ただし範囲が重なる可能性はある (例: sg.c=0, mc1=3, mc2=5 → 列 0-3 と 0-5 で 0-3 が重なる)

**結論: L-bend を残したまま W/E 行共有にするのが最もシンプル。**
L-bend の H 重なりは「同じターミナルからの複数ポートは異なるチャネル行」で回避。

### 最終設計

1. **mkChG を変更**: E チャネルの行を W チャネルと同じにする (ch + 1)。S チャネルの列を N チャネルと同じにする (ch + 1)。
2. **グリッドサイズ**: nR = 1 + max(pW,pE) + mH + 1。nC = 1 + max(pN,pS) + mV + 1。
   (pW+pE ではなく max(pW,pE) に戻る)
3. **衝突回避**: L-bend は sg/dg のチャネル行/列を使う。同じターミナルからの複数ポートは異なるチャネルなので異なる行/列。Z/U-bend は専用中間チャネルで保証。
4. **L-bend の H 重なり確認**: 
   Port A (W[a]→N[b]): H at row_a, cols 0..col_b
   Port B (W[a]→N[c]): H at row_a2, cols 0..col_c (a の別チャネル)
   row_a ≠ row_a2 → OK ✓
   Port C (E[d]→N[b]): H at row_d, cols col_b..nC-1. row_d ≠ row_a (d ≠ a ならば) → OK
   同じ idx (a=d) でも W と E は同じ行を共有するが、ポート A の sg=W[a] とポート C の sg=E[a] が同じ行。H が重なる可能性:
   - Port A H: row_a, cols 0..col_b
   - Port C H: row_a, cols col_b..nC-1
   → 列 col_b で隣接するだけで重ならない ✓ (Port A は col_b まで、Port C は col_b から)
   
   ただし Port A H 範囲と Port C H 範囲が col_b で接触。同じ行で H が繋がる。
   これは「重なり」ではなく「接触」→ 同じセル (row_a, col_b) に H が 2 回配置される。
   → placeH が同じセルに 2 回 H を置く → hOwner conflict として検出される。
   
   **問題**: W[a] と E[a] が同行のとき、L-bend の H 範囲が col_b で重なる。
   
   **対策**: 同一ターミナル idx の W と E が同行にいる場合、H 範囲の重なりを確認:
   - W[a]→N[b] の H: cols 0..col_b
   - E[a]→N[c] の H: cols col_c..nC-1
   - col_b < col_c なら重ならない (L-bend 同士の折れ点が異なる列)
   - col_b = col_c のとき: 同じ列で H が接触。ただし一方は cols 0..col_b、他方は cols col_b..nC-1。
     col_b のセルに両方の H が入る → 重なり!
   
   **修正**: col_b = col_c (同じ N チャネル列を折れ点として共有) のケースで問題。
   これは W[a]→N[b] と E[a]→N[b] (同じ N ターミナル宛) の場合に発生。
   → N[b] のチャネルは 2 つのポートで異なるチャネル番号が割り当てられる
   → col_b ≠ col_c ✓
   
   **結論**: 同じターミナルへの複数ポートは異なるチャネル (異なる列) を持つので重ならない ✓

## 実装変更

ルーティング (routeBlockPorts) は一切変更しない。表示レイヤのみ変更。

### 1. termPos の位置合わせ

termPos マップで E/S ターミナルの座標を W/N に合わせる:

```
termPos['E' + i] = { x: cellSize, y: termPos['W' + i].y }  // E は W の y を使う
termPos['S' + i] = { x: termPos['N' + i].x, y: cellSize }  // S は N の x を使う
```

ポートに参加していない (デフォルト位置の) ターミナルも同様:
W[i] と E[i] は同じ idx なら同じ y、N[i] と S[i] は同じ idx なら同じ x。

### 2. スパインのブリッジ描画

E ターミナルのスパイン: E チャネル位置 (ルート終点) から termPos['E'+i] (= W[i].y) への
V ブリッジ線を描画。ルート終点とブロック境界を接続する。

S ターミナルのスパインも同様: S チャネル位置から termPos['S'+i] (= N[i].x) への
H ブリッジ線を描画。

### 3. statePos

現行の E→W, N→S 座標変換ロジック (epic 1 で実装済み) をそのまま使用:
- E 型: `gridTermPos['W' + idx].y` を使用
- N 型: `gridTermPos['S' + idx].x` を使用 (S は N に合わせ済みなので結果的に N の x)

### 4. nx/ny ポート描画

- nx: `gridTermPos['W' + idx].y` を使用 (変更なし)
- ny: `gridTermPos['S' + idx].x` を使用 (S = N の x なので正しい)
