# Epic 1: テスト仕様書

## 概要

Epic 1 (Viewer ポート結線の修正) の 2 つの Issue に対するテスト仕様。
対象ファイル: `index.html` (Repeated Maze Visualizer v0.2)

## テスト環境

- ブラウザ: 最新版 Chrome / Firefox / Edge いずれか
- ローカル HTTP サーバ経由で `index.html` を開く
- 表示モード: **undirected** (directed チェックボックス OFF)
- x,y only チェックボックス: OFF (normal ビュー)

## テストデータ

### Maze 入力

```
normal: W0-E2, W2-E3, W3-S0, N0-W0, E4-W5, W5-N1, S1-W4, W6-E7, W7-E8, W8-S2, N2-W6, E9-W10, W10-N3, S3-W9, W11-E12, W12-E13, W13-S4, N4-W11, E14-W15, W15-N5, S5-W14, W16-S6, N6-W16, S7-W4, S8-W9, S9-W14, S10-W1; nx: E4-E6, E9-E11, E14-E16; ny: N0-N7, N2-N8, N4-N9, N6-N10
```

### Answer 入力 (パス検証用)

テストデータの迷路を solver で解いたパス、または手動で作成した
隣接ブロック間を跨ぐパスを使用する。
例:
```
(0,1,E0) -> (1,1,N0) -> (0,2,E2) -> (1,1,N1) -> (0,1,E1)
```

## テストケース

### TC-1: normal block の orthogonal ルーティング

| 項目       | 内容 |
|-----------|------|
| 対象 Issue | 共通前提 |
| 対象関数   | `routeBlockPorts()`, `drawPolyline()` |
| 前提条件   | テストデータの Maze を入力し Draw Maze ボタンを押す (undirected モード) |

#### 確認手順

1. normal block (bx >= 1, by >= 1) 内のポート結線を目視確認する。
2. 以下のポートが描画されていることを確認する:
   - W0-E2, W2-E3, W3-S0, N0-W0
   - E4-W5, W5-N1, S1-W4
   - W6-E7, W7-E8, W8-S2, N2-W6
   - E9-W10, W10-N3, S3-W9
   - W11-E12, W12-E13, W13-S4, N4-W11
   - E14-W15, W15-N5, S5-W14
   - W16-S6, N6-W16
   - S7-W4, S8-W9, S9-W14, S10-W1

#### 合格基準

- [ ] 全ポートが **直交折れ線** (水平・垂直セグメントのみ) で描画される
- [ ] 結線同士が **不正に重なっていない** (交差は許容、平行重複は不可)
- [ ] 各結線の両端がブロック辺上の正しいターミナル位置に到達している
- [ ] サブターミナル (同一ターミナルに複数ポートが接続) のスパイン・ジャンクション点が正しく描画される

### TC-2: nx block ポートのグリッド整合 (Issue 1)

| 項目       | 内容 |
|-----------|------|
| 対象 Issue | Issue 1: nx/ny ポートが接続先ターミナルに繋がっていない |
| 対象関数   | `routeBlockPorts()` (termPos), nx 描画ロジック (L.760-789) |
| 前提条件   | TC-1 と同じ |

#### 対象ポート

| nx ポート | 対応する W ターミナル (隣接 normal block) |
|----------|----------------------------------------|
| E4-E6   | W4@(1,by) — W6@(1,by) |
| E9-E11  | W9@(1,by) — W11@(1,by) |
| E14-E16 | W14@(1,by) — W16@(1,by) |

#### 確認手順

1. bx=0 列の各ブロック (nx block) を確認する。
2. 各 nx ポート (E4-E6, E9-E11, E14-E16) が曲線で描画されていることを確認する。
3. 曲線の両端点が、隣接 normal block (bx=1) の対応する W ターミナルの
   **グリッド y 座標** と一致していることを確認する。

#### 合格基準

- [ ] E4-E6 曲線の端点 y 座標が、normal block 内の W4, W6 のグリッド y 座標にそれぞれ一致する
- [ ] E9-E11 曲線の端点 y 座標が、normal block 内の W9, W11 のグリッド y 座標にそれぞれ一致する
- [ ] E14-E16 曲線の端点 y 座標が、normal block 内の W14, W16 のグリッド y 座標にそれぞれ一致する
- [ ] 曲線端点が `termLocalPos` (等間隔配置) にフォールバックしていない

#### 不合格時の典型的な症状

- nx 曲線の端点と normal block のポート結線端点が y 方向にずれている
- `gridTermPos` にキーが存在しない場合、`termLocalPos` が使われ等間隔配置になる

### TC-3: ny block ポートのグリッド整合 (Issue 1)

| 項目       | 内容 |
|-----------|------|
| 対象 Issue | Issue 1: nx/ny ポートが接続先ターミナルに繋がっていない |
| 対象関数   | `routeBlockPorts()` (termPos), ny 描画ロジック (L.791-820) |
| 前提条件   | TC-1 と同じ |

#### 対象ポート

| ny ポート | 対応する S/N ターミナル (隣接 normal block) |
|----------|------------------------------------------|
| N0-N7   | S0@(bx,1) / N7 — S チャネル x 座標 |
| N2-N8   | S2@(bx,1) / N8 — S チャネル x 座標 |
| N4-N9   | S4@(bx,1) / N9 — S チャネル x 座標 |
| N6-N10  | S6@(bx,1) / N10 — S チャネル x 座標 |

#### 確認手順

1. by=0 行の各ブロック (ny block) を確認する。
2. 各 ny ポート (N0-N7, N2-N8, N4-N9, N6-N10) が曲線で描画されていることを確認する。
3. 曲線の両端点が、隣接 normal block (by=1) の対応する S (または N) ターミナルの
   **グリッド x 座標** と一致していることを確認する。

#### 合格基準

- [ ] N0-N7 曲線の端点 x 座標が、normal block 内の対応する N/S チャネルのグリッド x 座標にそれぞれ一致する
- [ ] N2-N8 曲線の端点 x 座標が、同様に一致する
- [ ] N4-N9 曲線の端点 x 座標が、同様に一致する
- [ ] N6-N10 曲線の端点 x 座標が、同様に一致する
- [ ] 曲線端点が `termLocalPos` (等間隔配置) にフォールバックしていない

#### 不合格時の典型的な症状

- ny 曲線の端点と normal block のポート結線端点が x 方向にずれている
- ポートに参加していないターミナル (例: N7, N8, N9, N10) が `gridTermPos` に無く、
  `termLocalPos` が使われる

### TC-4: Answer パスの描画 (Issue 2)

| 項目       | 内容 |
|-----------|------|
| 対象 Issue | Issue 2: Answer パスが斜め線で描画される |
| 対象関数   | `statePos()`, パス描画ロジック (L.843-876) |
| 前提条件   | テストデータの Maze + Answer を入力し Draw Maze ボタンを押す |

#### 確認手順

1. Answer 欄にパスを入力する。
2. Draw Maze ボタンを押す。
3. パスの各ステップ間の赤い線を確認する。
4. 隣接ブロック間を跨ぐパスセグメントが **水平または垂直** であることを確認する。

#### 合格基準

- [ ] 隣接ブロック間の E→W パスセグメントが **水平線** になる (y 座標が一致)
- [ ] 隣接ブロック間の N→S パスセグメントが **垂直線** になる (x 座標が一致)
- [ ] ステップ番号円がグリッドのターミナル位置に乗っている
- [ ] **斜め線が一切表示されない** (ブロック内での移動を除く)

#### 不合格時の典型的な症状

- ブロック (bx, by) の E[i] と ブロック (bx+1, by) の W[i] が異なる y に描画され、
  斜めの赤い線になる
- `statePos` が `gridTermPos` にキーを見つけられず `termLocalPos` にフォールバックする

### TC-5: Start/Goal マーカーのグリッド整合

| 項目       | 内容 |
|-----------|------|
| 対象 Issue | Issue 1, Issue 2 共通 |
| 対象関数   | `statePos()`, Start/Goal 描画ロジック (L.823-839) |
| 前提条件   | TC-1 と同じ |

#### 確認手順

1. Start マーカー (緑) と Goal マーカー (赤) の位置を確認する。
2. Start = E0@(0,1), Goal = E1@(0,1) なので、bx=0 列の nx block 右端に表示される。
3. マーカーの y 座標が、隣接 normal block (bx=1, by=1) の W0, W1 の
   グリッド y 座標と一致することを確認する。

#### 合格基準

- [ ] Start マーカー (E0) の y 座標が W0 のグリッド y 座標と一致する
- [ ] Goal マーカー (E1) の y 座標が W1 のグリッド y 座標と一致する
- [ ] マーカーが nx ポート曲線や normal block ポート結線の端点と同じ高さに表示される

### TC-6: gridTermPos の全ターミナル網羅 (ユニットレベル)

| 項目       | 内容 |
|-----------|------|
| 対象 Issue | Issue 1 の根本原因 |
| 対象関数   | `routeBlockPorts()` 戻り値の `termPos` |
| 前提条件   | テストデータの normal ポートで `routeBlockPorts()` を呼び出す |

#### 確認手順

ブラウザの DevTools コンソールで以下を確認する:

1. `routeBlockPorts()` の戻り値 `termPos` を取得する。
2. 0 から nterm-1 までの全 idx について、以下のキーが存在するか確認する:
   - `W0` .. `W{nterm-1}`
   - `E0` .. `E{nterm-1}`
   - `N0` .. `N{nterm-1}`
   - `S0` .. `S{nterm-1}`

#### 合格基準

- [ ] **全ターミナル** (W/E/N/S x 0..nterm-1) が `termPos` に含まれる
- [ ] ポートに参加していないターミナルにも `termLocalPos` 相当のデフォルト座標が割り当てられている
- [ ] ポートに参加しているターミナルの座標が、グリッドルーティングの実際のチャネル位置と一致する

#### 検証コード例

```javascript
// DevTools Console で実行
// Draw Maze 実行後、gridResult を参照できるようにする必要あり
// (デバッグ用に window.gridResult = gridResult; を一時的に追加)

const tp = window.gridResult.termPos;
const nterm = /* テストデータの nterm */;
const missing = [];
for (let i = 0; i < nterm; i++) {
  for (const d of ['W', 'E', 'N', 'S']) {
    if (!tp[d + i]) missing.push(d + i);
  }
}
console.log('Missing terminals:', missing);
// 合格: missing === []
```
