# diagonal2 アルゴリズム レビューレポート

## 概要

diagonal2.md で提案された決定論的衝突回避ルーティングアルゴリズムを、
team-lead 指定の 5 項目について分析し、反例を探した。

**結論: 反例は見つからなかった。アルゴリズムの衝突回避証明は健全。**

ただし、実装上の注意点と、証明が依存する前提条件を以下に整理する。

---

## 1. L-bend 衝突: 同一ターミナルを共有する 2 つの L-bend

### 検証ケース: W0→N0 と W0→S1

**チャネル割当て:**
- W0 には 2 つのポート端: Port A の src (ch\_a)、Port B の src (ch\_b)
- ch\_a ≠ ch\_b (同一ターミナルでも sideCh[W]++ で別チャネルが割り当てられる)

**H セグメント:**
- Port A: row = ch\_a + 1 (W チャネル行), cols 0..col\_N\_ch\_c
- Port B: row = ch\_b + 1 (W チャネル行), cols 0..col\_S\_ch\_d
- **ch\_a ≠ ch\_b → 別の行 → H 重なりなし ✓**

**V セグメント:**
- Port A: col = col\_N\_ch\_c (N チャネル列), rows 0..row\_ch\_a
- Port B: col = col\_S\_ch\_d (S チャネル列), rows row\_ch\_b..S\_border
- N チャネル列 (1..pN) と S チャネル列 (pN+1..pN+pS) は**異なる列範囲** → V 重なりなし ✓

### 追加検証: W0→N0 と W0→N1 (同じ src 方向、同じ dst 方向)

- H: ch\_a ≠ ch\_b → 別行 ✓
- V: N0 と N1 は別ターミナル → 別 N チャネル → 別列 ✓

### 追加検証: W0→N0 と W1→N0 (異なる src、同じ dst)

- H: W0 と W1 は別ターミナル → 別 W チャネル → 別行 ✓
- V: N0 に 2 つの dst ポート → ch\_c ≠ ch\_d → 別列 ✓

**判定: 反例なし。チャネル一意性により L-bend 間の衝突は不可能。**

---

## 2. L-bend + Z-bend が共通の dst ターミナルを共有するケース

### 検証ケース: W0→N0 (L-bend) と E1→N0 (L-bend)

注意: E と N は隣接辺なので E1→N0 も **L-bend**（Z-bend ではない）。

- Port A: V at col\_N\_ch\_a (N0 の dst チャネル)
- Port B: V at col\_N\_ch\_b (N0 の dst チャネル)
- ch\_a ≠ ch\_b (N0 に 2 つの dst ポート) → 別列 ✓
- H: Port A は W チャネル行、Port B は E チャネル行 → 別行域 ✓

### 検証ケース: W0→N0 (L-bend) と S0→N0 (Z-bend, !isLR)

Z-bend の構成:
- V1: col\_S\_ch\_c (S チャネル列), rows S\_border..mr
- H: row mr (専用 H 中間行), cols col\_S\_ch\_c..col\_N\_ch\_d
- V2: col\_N\_ch\_d (N チャネル列), rows mr..0

L-bend の V: col\_N\_ch\_a, rows 0..row\_W\_ch\_e
Z-bend の V2: col\_N\_ch\_d, rows mr..0

**ch\_a ≠ ch\_d** (N0 に 2 つの dst ポート → 別チャネル) → 別 N 列 ✓

Z-bend の V は dedicated middle column を使うか？
→ **いいえ**。S0→N0 の V1 は S チャネル列 (src 列)、V2 は N チャネル列 (dst 列)。
中間セグメントは H (dedicated H 中間行)。
→ V1 は S チャネル域、V2 は N チャネル域、L-bend の V も N チャネル域。
→ V2 と L-bend の V: 同じ N チャネル域だが ch\_a ≠ ch\_d → 別列 ✓

**判定: 反例なし。L-bend の V と Z-bend の V は同じ列域に入りうるが、チャネル一意性で保護される。**

---

## 3. 同辺ポート (W0→W1, W0→W2)

### チャネル割当て

Port A: W0→W1
- src W0 → W ch\_a
- dst W1 → W ch\_b

Port B: W0→W2
- src W0 → W ch\_c (ch\_c ≠ ch\_a: 同一ターミナル W0 でも別チャネル)
- dst W2 → W ch\_d

全チャネル ch\_a, ch\_b, ch\_c, ch\_d は一意 (sideCh[W]++ による連番割り当て)。

### ルーティング

Port A (Z-bend, isLR):
- H1: row ch\_a, cols 0..mc\_A (dedicated V 中間列)
- V: col mc\_A, rows ch\_a..ch\_b
- H2: row ch\_b, cols mc\_A..0

Port B (Z-bend, isLR):
- H1: row ch\_c, cols 0..mc\_B (別の dedicated V 中間列)
- V: col mc\_B, rows ch\_c..ch\_d
- H2: row ch\_d, cols mc\_B..0

**H 重なり**: ch\_a, ch\_b, ch\_c, ch\_d は全て異なる W チャネル行 → 4 つの H は全て別行 ✓
**V 重なり**: mc\_A ≠ mc\_B (ポートごとに専用中間列) → 別列 ✓

### ch\_src ≠ ch\_dst の保証

W0→W1 の場合: src=W0, dst=W1 → 異なるターミナル → 当然異なるチャネル ✓
W0→W0 は自己ループで排除済み ✓

**仮に** 同じターミナルの src/dst があったとしても (例: ポート A の src=W0 と ポート B の dst=W0)、
それらは**異なるポート端**なので sideCh[W]++ により異なるチャネルが割り当てられる。

**判定: 反例なし。同辺ポートでもチャネル一意性と専用中間列で衝突しない。**

## 4. グリッドサイズ: 20 ポート、nterm=8 での評価

### 計算

- 20 ポート → 40 ポート端 (各ポートが src + dst)
- pW + pE + pN + pS = 40 (全ポート端の合計)
- mH ≤ 20 (各 Z/U-bend で最大 1 中間行)
- mV ≤ 20 (各 Z/U-bend で最大 1 中間列)

**最大グリッドサイズ:**
- nR = 1 + pW + pE + mH + 1 ≤ 1 + 40 + 20 + 1 = 62
- nC = 1 + pN + pS + mV + 1 ≤ 1 + 40 + 20 + 1 = 62

実際には pW+pE と pN+pS は合計 40 を分け合うので、バランスした場合:
- pW+pE ≈ 20, pN+pS ≈ 20
- nR ≈ 1 + 20 + 10 + 1 = 32
- nC ≈ 1 + 20 + 10 + 1 = 32

**比較: 現行アルゴリズム (v0.8)**
- 初期グリッド: pW+pE+2 × pN+pS+2 (約 22×22)
- expansion 後: 失敗ケースで 50+ 回の insertRow/insertCol → 70×70 以上に膨張

**評価**: 新アルゴリズムのグリッドは現行の expansion 前より大きいが、
expansion 後よりは小さい。事前に確定するので膨張の心配がない。
cellSize=240 の場合、セルサイズ = 240/32 ≈ 7.5px。十分な描画解像度。✓

**20 ポート全て Z/U-bend の最悪ケースでも 62×62 ≈ 3.9px/cell。
視認性は低下するが機能上の問題はない。**

**判定: グリッドサイズは妥当。現行の不確定膨張より改善。**

## 5. サブターミナルスパインとルートセグメントの干渉

### スパインの構造

W0 ターミナルに 3 ポートが接続している場合:
- 3 チャネル (ch\_0, ch\_1, ch\_2) → 3 つの行 (row\_0, row\_1, row\_2)
- スパイン:
  - 水平スタブ: ブロック辺 (col 0) から最初のチャネルの x 位置まで
  - 垂直スパイン: 最初のチャネルから最後のチャネルまで

スパインはピクセル座標で描画され、**グリッドの H/V フラグには含まれない**。
ルーティングの衝突検出対象外。

### 視覚的重なりの可能性

スパインの垂直部分は、チャネル列の位置 (col 0.5 * cellWidth/nC) に描画される。
ルートの H セグメントも同じ列を通過しうる。

しかし:
- スパインは col 0 (W 境界) のピクセル位置に近い場所に描画
- ルートの H セグメントはグリッドセルの中心に描画
- ルーティング上の重なり (hOwner/vOwner conflict) とは無関係

**判定: スパインはルーティングの衝突検出対象外。視覚的には重なる場合があるが、
これは現行アルゴリズムでも同様であり、仕様上の問題ではない。**

## 証明が依存する前提条件

アルゴリズムの正しさは以下の前提に依存する:

1. **チャネル一意性**: 同一ターミナルの異なるポート端は異なるチャネル番号を取得する。
   → 現行の `sideCh[dir]++` による連番割り当てで保証 ✓

2. **自己ループ排除**: src と dst が同一ターミナル (例: W0→W0) のポートは存在しない。
   → ポート生成ルールで排除 ✓

3. **中間チャネルの一意割り当て**: 各 Z/U-bend ポートが専用の中間行/列を取得する。
   → 連番割り当てで保証。同じ中間行/列を 2 ポートが共有しない ✓

4. **行/列範囲の非重複**:
   - W チャネル行 (1..pW), E チャネル行 (pW+1..pW+pE), H 中間行 (pW+pE+1..pW+pE+mH) → 範囲が重複しない ✓
   - N チャネル列 (1..pN), S チャネル列 (pN+1..pN+pS), V 中間列 (pN+pS+1..pN+pS+mV) → 範囲が重複しない ✓

5. **L-bend の隣接辺判定が正しい**: W-N, W-S, E-N, E-S が隣接辺であること。
   → 定義通り ✓

## まとめ

| 検証項目 | 結果 | 反例 |
|---------|------|------|
| 1. L-bend 同士 (共通ターミナル) | 衝突しない | なし |
| 2. L-bend + Z-bend (共通 dst) | 衝突しない | なし |
| 3. 同辺ポート | 衝突しない | なし |
| 4. グリッドサイズ | 妥当 (最大 62×62) | N/A |
| 5. スパイン干渉 | 検出対象外、問題なし | N/A |

**反例は見つからなかった。** アルゴリズムの衝突回避は、チャネル一意性と行/列範囲の分離に基づき、
原理的に保証される。

### 衝突不可能の根拠 (要約)

全ての H セグメントは以下のいずれかの行にのみ存在:
- W チャネル行 (範囲 1..pW): ポート端ごとに一意
- E チャネル行 (範囲 pW+1..pW+pE): ポート端ごとに一意
- H 中間行 (範囲 pW+pE+1..pW+pE+mH): ポートごとに一意

これら 3 範囲は互いに重複しない。
→ **異なるポートの H セグメントが同一行を共有することは不可能**
→ H-H 重なりは発生しない

V セグメントについても同様 (N チャネル列、S チャネル列、V 中間列が非重複)。
→ **V-V 重なりも発生しない**

H+V (交差) は仕様上許容。 **QED.**

### 実装上の注意

1. **L-bend の折れ点**: `isLR(src)` なら `(sg.r, dg.c)`、`!isLR(src)` なら `(dg.r, sg.c)`。
   この判定を間違えると衝突する。

2. **mH/mV のカウント**: L-bend (隣接辺) は中間セグメント不要。
   Z/U-bend のみをカウントすること。isLR→mV++, !isLR→mH++。

3. **termPos 構築**: グリッドレイアウトが変わるので、statePos や nx/ny 描画の
   ピクセル座標計算もこの新レイアウトに合わせる必要がある。

4. **最小グリッドサイズ**: 0 ポートで nR=2, nC=2 になるが、空ケースは別処理済み。
   1 ポート (L-bend) で nR=3, nC=3 → 十分。
